# 分群管理员权限设计

## 目标

管理员权限按 QQ 群隔离。QQ群主和群管理员自动拥有当前群的 bot 管理权限；普通成员只能通过当前群的手动授权获得权限。

## 权限数据

重构 `admins` 表并放弃兼容现有的全局授权数据。表使用 `(group_id, user_id)` 联合主键，保存以下状态：

- `manual_granted`：是否由当前群有权限的用户手动授权。
- `qq_role`：最近一次从 NapCat 查询到的 `owner`、`admin` 或 `member`。
- `created_at`、`updated_at`：记录创建和最近同步时间。

手动授权与 QQ 群角色相互独立。一个已经被手动授权的成员成为群管理员后，`manual_granted` 仍然保留；将来被降为普通成员时仍可依靠原有手动授权操作 bot。

## 鉴权流程

除仅显示帮助的空 `/admin` 外，每次管理员命令都执行以下流程：

1. 通过 NapCat `get_group_member_info` 查询执行者在当前群的实时角色。
2. 将角色写入当前群对应的管理员记录；角色变为普通成员且没有手动授权时删除无效记录。
3. `owner` 或 `admin` 直接通过鉴权。
4. `member` 仅在当前群的 `manual_granted` 为真时通过鉴权。
5. NapCat 角色查询失败时拒绝本次操作并返回可理解的临时错误，不使用数据库中的旧角色放行。

消息事件自带的 sender role 只作为消息数据保留，不代替实时查询。

## 管理命令

- `/admin 添加管理员 @用户`：查询目标在当前群的实时角色。普通成员设置当前群的 `manual_granted=true`；群主或群管理员无需重复授权，返回其权限来自 QQ 群角色。
- `/admin 移除管理员 @用户`：查询目标实时角色。群主或群管理员不可移除；普通成员清除当前群的手动授权。
- `/admin 移除所有管理员`：只清除当前群所有手动授权。已记录的群主和群管理员角色保留。
- `/admin 所有管理员`：只列出当前群已记录的管理员，并标明 QQ 群主、QQ 群管理员或手动授权来源。

任何群的命令都不能读取、增加、移除或清空其他群的手动授权。

## 组件边界

- NapCat sender 提供按 `groupID` 和 `userID` 查询群成员角色的接口。
- 命令路由负责在执行管理员命令前查询执行者角色，并在添加或移除授权时查询目标角色。
- `AdminHandler` 负责权限判定、角色同步和管理员子命令语义。
- `AdminStore` 的所有管理员读写接口必须包含 `groupID`。
- MySQL schema、GORM model 和生成的 query 同步更新为新的分群结构。

## 错误处理

- 无法查询执行者角色：不执行命令，提示“暂时无法确认群身份，请稍后重试”。
- 无法查询目标角色：不修改授权，提示“暂时无法确认该成员身份，请稍后重试”。
- 数据库读写失败：不返回成功提示，由现有日志链路记录具体错误。
- 目标不是当前群成员时：按 NapCat 查询失败处理，不创建跨群或孤立授权。

## 测试

- 群主和群管理员无需手动授权即可执行命令，并同步角色到当前群记录。
- 普通成员只能依靠当前群的手动授权，另一个群中的同一 QQ 号不继承权限。
- 群角色降级后实时失权；存在独立手动授权时仍然保留权限。
- 群主和群管理员不能被单个移除或批量清除。
- 添加、移除、列出和清空手动授权只影响当前群。
- NapCat 查询执行者或目标角色失败时不执行受保护操作。
- MySQL 存储使用 `(group_id, user_id)` 隔离读写。

## 部署约束

项目仍采用 schema-first，不增加运行时自动迁移。开发部署需要按新的 `deploy/mysql/init/001_schema.sql` 重建或手动替换 `admins` 表。
